Rule 90 is one of 256 possible rules (numbered 0 through 255). The rule number is a compact way to encode the 8 outputs: write them in order from neighborhood 7 down to 0, and you get an 8-bit binary number. Rule 90's outputs 01011010 equal 90 in decimal.

The method below works for any rule number. Pass in a neighborhood and a rule number, and it returns the correct output. This uses binary math that is not on the AP exam; you don't need to understand how it works internally. What matters is that you can call it with any rule number from 0 to 255 and it applies that rule correctly.

int applyRule(int left, int center, int right, int ruleNum) {
    int[] ruleOutputs = new int[8];
    for (int i = 0; i < 8; i++) {
        ruleOutputs[i] = (ruleNum >> i) & 1;
    }
    int neighborhoodIndex = left * 4 + center * 2 + right;
    return ruleOutputs[neighborhoodIndex];
}
How it works

Step 1: build the output array from the rule number. ruleOutputs is an array of 8 values, one for each possible neighborhood. These are the same 0s and 1s you see in the "next value" column of the rule table. The loop fills this array by unpacking the rule number, one digit at a time.

Step 2: convert the neighborhood to an index (0 to 7). The three cells are treated as digits of a number: left is worth 4, center is worth 2, right is worth 1. Adding them gives a unique index for each of the 8 possible neighborhoods.

For example, the neighborhood left=1, center=0, right=1 gives:

1×4 + 0×2 + 1×1 = 5

That's position 5 in the table, the row where left is alive, center is dead, right is alive.

Step 3: look up the answer. ruleOutputs[neighborhoodIndex] returns the output for this neighborhood, the same value you would read from the table by hand.

How the bit unpacking works

Every integer is stored as a sequence of binary digits. The number 90 in binary is 01011010. Each of those 8 digits is one output value for Rule 90, matching the 8 rows of the "next value" column.

The expression (ruleNum >> i) & 1 pulls out digit number i. The >> shifts all the digits right by i positions, so digit i lands in the last slot. Then & 1 keeps only that last digit, giving 0 or 1.

For Rule 90 and i = 3: the binary digits are 0 1 0 1 1 0 1 0. Digit 3 (counting from the right, starting at 0) is 1. So ruleOutputs[3] = 1, meaning neighborhood index 3 produces a live cell. That matches the table row where left=0, center=1, right=1.

2. A student writes this method to compute a "smoothed" version of an array, where each element is replaced by the average of itself and its two neighbors:

void smooth(int[] arr) {
    for (int i = 1; i < arr.length - 1; i++) {
        arr[i] = (arr[i - 1] + arr[i] + arr[i + 1]) / 3;
    }
}

The method produces incorrect results. What is the most likely cause?

Choose one:

A) The loop bounds are wrong; it should start at 0 and go to arr.length.
The bounds are actually correct for neighbor access. Starting at 0 would cause an out-of-bounds access on arr[i - 1].
B) Integer division truncates the result, so averages are always rounded down.
That's a real issue with integer division, but it's not what makes the results incorrect in terms of which values get used. There's a more fundamental problem here.
C) The method updates arr[i] in place, so earlier cells have already been changed when later cells read them as neighbors.
Correct. By the time the loop reaches index 3, arr[2] has already been overwritten with a smoothed value, not the original. The fix is to write results into a separate array, exactly like makeNextGen.
D) The method has no return statement, so the changes are lost.
Arrays are passed by reference in Java. Changes made to arr inside the method affect the original array; no return statement is needed for the writes to take effect.
