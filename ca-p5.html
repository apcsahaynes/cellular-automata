Cellular Automata

Page 5 — Game of Life

On this page, you'll build on what you built on page 4 to implement Conway's Game of Life, the most well known cellular automata and one of the most famous examples of emergence in computing.

Before you start: Read the Conway Life Wiki introduction to see what Game of Life looks like running and why it's interesting. Pay attention to what "stable," "oscillating," and "moving" patterns mean.

Game of Life is a 2D automaton where every cell is either alive (1) or dead (0), and four rules decide the next generation. All cells update simultaneously:

conditionresult
live cell, fewer than 2 live neighborsdies (underpopulation)
live cell, 2 or 3 live neighborssurvives
live cell, more than 3 live neighborsdies (overpopulation)
dead cell, exactly 3 live neighborsbecomes alive (reproduction)

Every other case produces a dead cell. You can state the survival cases more compactly: a live cell survives with 2 or 3 neighbors; a dead cell is born with exactly 3.

Hint: working through the blinker

Consider two cells. First, the middle cell (row 2, col 2 if you start from 0). Its neighbors: above it is alive, below it is alive, left and right are dead, and all four diagonals are dead. That's 2 live neighbors, so it survives.

Now take a dead cell at row 2, col 1 (just left of the middle). Its neighbors include the three blinker cells. Count them: that's 3 live neighbors, so it become alive.

Make sure you skip the outer border, which means all neighbor accesses are guaranteed in bounds. Edge cells are effectively always dead, which is fine for exploring patterns in the interior.

Hint

Two conditions result in a live next cell. Write them as two separate if statements or combine them with ||:

  • The cell is currently alive (currentGrid[r][c] == 1) and has 2 or 3 live neighbors
  • The cell is currently dead (currentGrid[r][c] == 0) and has exactly 3 live neighbors

Since nextGrid starts as all zeros, you only need to write a 1 when a cell should be alive.
